Nanite遵循的设计理念是——三角形的绘制数量超过像素数量就是一种浪费。

可以看到Nanite是基于GPU渲染管线编写的。GPU Driven Pipeline把模型的顶点数据进一步切分为更细粒度的单位Cluster。每个Cluster的精细度可以更好地适应顶点处理阶段的Cache大小，并进行各类剔除Culling。

Nanite虚拟几何体系统：系统只会流送绘制画面必须的数据，只有这些会存在内存里。换言之只需要渲染你可以看见的物体，不会绘制看不见的部分。

前向工作：虚拟纹理系统

  • 古代山谷demo中的任意一个megascans资产，都不会使用独特的法线贴图和粗糙度贴图。使用了电影级的高品质主网格体（cinematic high master mesh），借助Quixel Bridge拖进UE中。
    • 每个资产使用的是细节平铺的方式（detail tiling）
    • Nanite的普适性
    • 扫描技术的提高，不需要法线贴图来驱动细节
  • Nanite技术是一个备选项。资产的精细程度可以自己决定。
  • 正在构建中的一种资产类型：Mega Assemblies，通过将megascan资产组合成更大的资产，用于快速布置于构建场景。

使用Nanite构建相较于地形构建的优点：

  • 三角形密度
  • 静态网格体——不仅仅是高度图height map可以解决的问题
    • 像素深度偏移的问题？不需要使用，因为资产本身就是高度细分，结合非常好，不需要soft blend
    • 原因：很好地做到遮蔽剔除fine grain occlusion culling，并且不占用资源
  • 多边形数量的预算、绘制调用的预算、网格体相关内存预算都可以不用考虑。不需要一系列复杂的资源和质量的平衡。

探究2：过度绘制Overdraw

在Nanite可视化界面中打开Overdraw可视化，可以看到Nanite在剔除方面做的努力。在这个热度图中，颜色越亮的地方，Nanite尝试绘制的次数越多，颜色越暗的地方，Nanite做的剔除越多，过度绘制也就越少。

可以看到在这几个资产重合的地方，进行了大量的剔除优化工作，使得接缝处的过度绘制大大减少

过度绘制发生的很大的情况是，表面彼此之间非常接近（堆叠问题），比如地面附近。这会导致很大的开销。

遮挡剔除的工作流程：找到几何体边界的资产、三角形的群集（cluster），然后测试是否课件。如果是在掠射角（glancing angle）的情况下，这些几何体的堆叠构成表面。这些几何体因为比较平坦（如地面部分的资产），因而在堆叠起来的时候表面之间的距离非常近。在从掠射角观察的角度来看，Nanite很难判断这个几何体是不是位于另一个表面之下。这在高空视角下非常严重。

（左图在地面上，右图在高空中）

Nanite原理讲解

几何体比纹理要复杂的多。

需求：不会大幅度地修改原有的CG制作流程。支持导入任何mesh，仍然有UV和平铺的细节map，不会对纹理、材质、工具进行替换

入手点：三角形Triangles——图形学的一切基础

Nanite is a completely GPU driven pipeline.

  • Renderer now RETAINED mode. 在GPU内存中会有这个场景的完整版本。不会每帧上传，只有上传到GPU的数据才会被更改。所有的顶点、索引数据都会存储在单个资源中。
  • 每次在GPU上进行GPU实例剔除（GPU instance culling）和三角形光栅化（Triangle rasterization）
  • 可以在一次间接绘制调用中绘制只通过整个场景的深度，不用为每个物体做剔除。

三角形集群剔除Triangle cluster culling

  • 一组三角形面片构成集群cluster（如128个）。构建边界数据，让每个集群有一个边界框bounding
  • 基于边界框剔除集群。可以基于视锥或基于遮挡。

技术：从材质中分离可见性Decouple visibility from material

理解：决定每个像素的可见性，这也是深度缓冲光栅化所做的。因为光栅化的过程中切换着色器开销很大。

需要解决的问题：

  • 光栅化时尽量不要切换着色器
  • 不要针对材质进行大量的过度绘制
  • 使用深度预处理避免过度绘制
  • 稠密mesh中像素块pixel quad的低效

延迟材质Deferred material

  • 材质通道写到GBuffer
  • 对于非透明刚体，只需要一次绘制
    • 全GPU驱动
    • 不只是一个深度与通道
    • 每次只光栅化一个三角形，因为只有单个几何体通道

集群层次Cluster hierarchy

通过层级化的cluster，可以定义LOD

像素缩放细节Pixel scale detail

  • 问题：三角形面片要多小来降低错误？
  • 需求：像素级大小的特点需要像素级大小的三角形来表示，才不会出现可见的错误

小三角形（像素大小）的问题：不适合通常的（硬件）光栅工具，因为高度平行，大量像素

软件光栅化可以达到3x速度

在这个demo中，大量的三角形是通过软件光栅化实现的。剩下的部分，比如较大的三角形，使用硬件光栅化工具。所以根据每个集群cluster来选择使用硬件光栅化或软件光栅化。

Nanite性能：上采样之前达到1400p，2.5ms剔除和光栅化，2ms基础通道，CPU时间几乎为0。在绘制材质时有极少的CPU开销，因为每次都有一次绘制调用。

Nanite数据：需要压缩。demo的Nanite数据最终的占用空间为6.14GB。（大量的数据是纹理素材）1M的三角面几何体在硬盘上的存储大约13.8MB（小于一个4K的法线贴图占用的空间）。每个三角大约占14.4B。

虚拟阴影贴图技术Virtual Shadow Map

  • 所有的虚拟阴影贴图都是16k*16k。这样一个纹理元素就会大致对应屏幕上的一个像素。使用VSM技术可以将纹理（要渲染的阴影）之渲染到屏幕上可见的阴影贴图像素上。Nanite会进行剔除，将LOD降低到需要的水平。同时支持缓存，避免过度绘制。
  • 支持多视图渲染，可以渲染每个光照和场景的阴影贴图。一次渲染场景中所有虚拟化的mipmap。可以做到硬影。使用SMRT滤波可以获得基于物理的混合阴影。

使用性

  • 首先聚焦于刚体几何体以及刚体的变形
  • 不支持非刚体物体及变形
  • 不支持骨骼动画
  • 不支持世界场景位置偏移
  • 不支持半透明或遮罩材质
  • 不支持曲面细分和置换
  • 不适合整合类的几何体：许多小型资产组合成一个多孔的体积，如树叶

7.18

补充学习一些针对于前几天视频上提到的内容

硬件光栅化与软件光栅化

  • 硬件光栅化：层次式Hierarchical。默认输入的三角形大小远远大于一个像素。

    • 第一步coarse raster，以一个三角形为输入，以8x8像素作为一个“块”，将三角形光栅化为若干个块。使用低分辨率的ZBuffer进行一次遮挡剔除。
    • 第二部fine raster，用来生成用于着色计算的像素，输出2x2的像素块pixel quad，以便于mipmap采样计算。

  • 软件光栅化：基于compute shader，适用于像素级的小三角形。

Visibility Buffer渲染管线

简单来讲是为了解决重绘问题。不再单独产生GBuffer，而是用带宽开销更低的Visibility Buffer来解决问题。

VBuffer的数据结构：（每个像素总共只需要存储8~12B）

  • InstanceID：像素所属实例
  • PrimitiveID：像素所属三角形
  • BarycentricCoord：像素所处三角形内的位置，用中心坐标表示
  • DepthBuffer：像素深度
  • MaterialID：像素所属材质

VBuffer渲染流程：

  • Visibility Pass：场景光栅化
  • Worklist Pass：屏幕划分，构建索引
  • Shading Pass：对材质ID进行软光栅，获取顶点属性并插值，进行表面着色

Nanite流程

  • streaming：传输，根据从上一帧回读的cluster page request数据，异步上传cluster渲染数据
  • init：初始化
  • cull：剔除
    • Instance Cull：实例化剔除
    • Persistent Cull：BVH和Cluster剔除，输出Cluster Buffer与Indirect参数
  • rasterize：光栅化
    • 硬件光栅化：使用vertex shader或primitive shader
    • 软件光栅化：使用compute shader，生成VBuffer
  • Post Persistent Cull
  • Post Rasterization
  • Emit Targets：生成渲染所需的深度相关数据与Buffer
  • Base Pass：根据VBuffer与Cluster相关Buffer生成延迟渲染所需GBuffer
  • 生成阴影所需深度
  • 回读在PersistentCull中产生的cluster page request
