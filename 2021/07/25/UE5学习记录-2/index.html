重要的是FInstanceDraw/FPersistentState/FStats的数据结构定义

实例化剔除过程负责执行对象实例级别的可见剔除，如果可见则将实例对应的BVH Root Node存储在CandidateNodesAndClusters中。当这个 Pass 结束后，CandidateNodesAndClusters中存储的是所有可见的实例的侯选BVH Node列表，列表中每个Node代表实例的 BVH Root Node。在接下来的PersistentClusterCull就可以根据这些根节点向下遍历执行剔除流程。

总结一下：Instance Cull以Mesh为单位，通过Instance Cull的Mesh会将其BVH的根节点送到Persistent Cull阶段进行层次式的剔除（若某个BVH节点被剔除，则不再处理其子节点）。

在代码中可以看到，实例化剔除主要包括视锥剔除和基于HZB的遮挡剔除。使用到的BoxCullFrustum()函数就是立方体与视锥相交检测函数。视截体裁剪的原理就是在3D渲染流水线开始之前就不让完全在视截体外的3D模型数据进入3D渲染流水线。

Persistent Culling 持续剔除（#1）

持续剔除是固定数量的计算线程，与场景的复杂性无关，而是可能需要检查某些空间结构是否存在遮挡。这一步执行 BVH Node 和 Cluster 剔除，输出 Rasterize 需要的 Cluster Buffer 以及 Indirect 参数。这部分源码在ClusterCulling.usf中，从PersistentClusterCull()入口。大致分为两个部分：

  • BVH节点剔除：每次循环使用线程组中的前8个线程获取有效的候选BVH Node，得出每组需要处理的BVH Node数量，通过NodeCull()函数处理候选节点的子节点。每个Nanite BVH最多有8个子节点，则每组64个线程可以并行处理64个BVH Node。

小结，别的一些东西的发现与学习

下一步要具体看看NodeCull与ClusterCull了！在这之前，注意到了这个主函数前有一大段UE的开发人员留下的代码注释，其实是很有启发性的！

这一部分涉及到了多线程处理的安排。系统需要接受的叶节点数量是动态的，小到0，大到数十万，因此将线程1:1地映射到BVH树上是不科学的——这将导致非常糟糕的串行处理。为了尽可能全面地使用GPU的多线程，UE5采用了Persistence Culling这一技术——持续剔除以及持久线程。从整个逻辑来看，Nanite流水线创造了很多工作线程来填充GPU工作区，保持其持续运行并按照一定的规则分配工作（进行NodeCull或者ClusterCull）。

这里遇到了一个名词——MPMC Job Queue，多生产者多消费者作业队列。典型的多生产者多消费者问题是生产者和消费者一一对应，有两个或多个进程，而缓冲区大小有限。一般来说对于小缓冲区而言可以不用设置专门的互斥变量，同步信号量也最多只有一个为1，因此任何时刻最多只有一个进程的P操作不会被阻塞。

在这样一个Nanite渲染的实际工作流中，新的作业被添加到作业队列中，供工作线程使用，并不产生新的线程。队列的初始化由前导的着色器过程生成。工作时，进程一边处理队列中的作业，一边产生新的作业（递归地遍历BVH树，将后续需要处理的节点添加到队列）。

最后一部分提到了the main motivations for handling the cluster culling in the same shader在同一个着色器中处理集群剔除的动机。这里留一个任务吧，下周看cluster culling的时候具体展开写一下。
