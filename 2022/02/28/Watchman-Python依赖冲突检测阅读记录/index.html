Watchman Python依赖冲突检测阅读记录

2022年02月28日 1.7k 字 大概 7 分钟

前言：这篇文章一方面是软件工程课程相关调研的论文，另一方面也是帮女票看看她的论文，在还有空的时候帮她力所能及地分担一些。祝她一切顺利~

论文题目：Watchman Monitoring Dependency Conflicts for Python Library Ecosystem

来源：2020 ICSE

主题：监测Python环境中的依赖冲突

摘要

问题：

  • RQ1（Manifestation Patterns表现模式）：DC问题如何在Python项目中表现出来？是否存在可用于自动诊断这些问题的常见模式？
  • RQ2（Fixing Strategies修复策略）：开发人员如何解决Python项目中的DC问题？是否有可用于自动修复这些问题的常见做法？

成果与意义：

  • 第一次在开源Python项目中对DC问题进行实证研究
  • 发布用于实证研究的数据集
  • 基于模型提出了一个形式化模型来模拟Python项目的构建过程，并开发了DC问题诊断技术

实证研究

RQ1

  • 发现1：235个DC问题中有211个（89.8%）涉及违反库版本约束，是由PyPI上的远程依赖项更新引起的。
  • 发现2：235个DC问题中有24个（10.2%）是由于远程依赖项与本地依赖环境中安装工具/库之间的冲突而产生的。
  • 发现3：将公共库的依赖关系限制为特定版本很容易导致下游项目出现的DC问题。
  • 发现4：当满足一个版本约束的库的安装版本接近另一个版本约束中指定的上限时，DC问题很容易发生。

RQ2

  • 策略1：调整直接依赖项的版本约束（98/235）。
  • 策略2：升级或降级需要冲突库的直接依赖项（27/235）。
  • 策略3：与上游项目协调，以调整冲突的版本限制（51/235）。
  • 策略4：删除冲突的直接依赖项并保留可传递的依赖项（8/235）。
  • 策略5：添加直接依赖项（16/235）。
  • 策略6：升级/降级开发工具（12/235）。
  • 策略7：创造一个独立的环境（8/235）。
  • 发现5：DC问题可能有多个修复程序。问题的表现形式、项目依赖关系图的拓扑结构、pip的安装规则以及上游项目和下游项目的版本约束之间的干扰可能会影响解决方案。

针对DC问题预测分析

通过FDG检测，对预防DC问题进行预测性分析

可能造成DC问题的情况：

  • 项目将依赖项限制为特定版本
    • A直接依赖于B，C直接依赖于A以及B，那么C就属于这种库
    • 这样的库越多，越接近某个预设值，越有可能出现DC问题
  • 库的安装版本接近版本约束中指定的上限

看图说话：

  • 在C2.0中，A的约束为=2.0。而C的下游项目B5.0则依赖于C2.0和A2.0。
    • C对于A的强约束条件可能导致与B的冲突。例如如果A更新了呢？
    • WATCHMAN会建议C的开发者放宽对A的约束条件
  • 对于T4.0有两条入边，一个来自X2.0，一个来自P4.0。
    • 假设前者在遍历时被先遍历到。X2.0对T的约束没有上限，那么就会安装T4.0
    • 后者对于T是有范围的限制要求的，那么如果PyPI中有T的新发布版本4.1，就会导致Q5.0的安装失败

讨论与展望

对结果合法性的威胁：关键词搜索可能会将一些无关的issue加入到研究的数据集，因此评估不一定准确。

局限性：

  • 只关注了导致build失败的DC问题。但是在一些情况下依赖冲突可能会导致Python项目中的语义不一致、运行时错误或其他后果。这个无法检测。
  • 预测分析中采用的规则只能寻找那两种type的子类，不饿能完全覆盖。
  • 无法检测所有实证研究中的DC问题模式。
