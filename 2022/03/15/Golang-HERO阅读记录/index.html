Golang HERO阅读记录

2022年03月15日 1.5k 字 大概 6 分钟

论文题目：HERO: On the Chaos When PATH Meets Modules

论文来源：ISCE 2021

主题：Go生态系统下两种不同的项目管理方式导致的Dependency Management问题与解决

背景

  • Go Modules与GOPATH不向后兼容。
  • 即使Golang项目及其引用的上游项目都使用Go Modules，也可能违反SIV规则。
  • 解决Golang项目的DM问题需要了解其上游和下游项目的最新信息，以及它们可能对两种库参考模式的异构使用。

SIV规则：必须有语义版本控制格式，以v2版本或更高的项目必须包含/v2后缀，而v0或v1版本则不应该在其模块或导入路径中包含版本后缀。

(a) lz4的v2.0.7迁移到了Go Modules，并声明模块路径github.com/pierrec/lz4/v2，然而他的下游项目仍在GOPATH，所以不识别/v2，引发了DM问题。因此lz4的v2.2.4移除了/v2后缀，解决了DM问题。但是对于已经迁移到Go Modules的下游项目出现了build错误，因为违反了SIV规则。最终lz4的v2.2.6又重用了GOPATH。

(b) go-i18n的v2.0.1为了更精细的库引用控件而使用了Go Modules，引发了下游使用GOPATH的项目的DM问题，同样是不识别/v2后缀。因此v2.0.2版本提供了一个额外的子目录go-il8n/v2，包含一份支持GOPATH的实现方法。

对于大多Golang项目包含两种：module-aware和module-unaware：

早期版本经常包含module-unaware的项目，特别是GOPATH环境下使用了DM管理工具，这些项目都是module unaware的，一旦迁移到Go Modules，就容易产生DM问题。

实证研究

RQ1（module-awareness的程度）

Golang生态中的项目的库参考模式迁移现状如何？它们在多大程度上是module-aware的？

Golang项目在迁移到Go Modules时面临挑战。截至2020年6月，GitHub上排名前2万的项目中只有35.9%迁移到了Go模块，排名前2万的项目中至少有51.0%是module-aware的。这两种库参考模式可能在生态系统中长期共存。

RQ2（问题的种类的诱因）

DM问题的常见类型是什么？其根本原因是什么？

DM问题通常是由于GOPATH和Go Modules的异构使用而产生的。它们的表现有两个共同的根本原因：（1）GOPATH和Go Modules以不同的方式解释导入路径；（2）Golang生态系统中没有严格执行SIV规则。

RQ3（解决方案）

解决DM问题的常见做法是什么？它们如何影响Golang生态系统？

  • GOPATH项目迁移到Go Modules
  • Go Modules项目回滚到GOPATH
  • 将Go Modules中v2+项目的发布策略从主要分支更改为子目录
  • 在下游项目的Vendor目录中的Go Modules中维护v2+库，而不是通过虚拟导入路径引用它们
  • 使用带有版本信息的replace指令避免在引用库中使用导入路径
  • 更新已更改存储库的库的导入路径
  • Go Modules中的项目修复配置项以严格遵守SIV规则
  • 使用特定版本的hash commit ID替换库引用中有问题的版本号

我们观察到针对DM问题的八种常见修复方案，覆盖了95.4%的研究问题。大多数解决方案可能会影响生态系统中的其他项目。在解决DM问题时，开发人员应该在好处和可能的后果之间进行权衡。

HERO

检测算法

  1. 收集当前项目信息：利用Github的APIrepository_url，通过查找是否有go.mod文件来查询当前项目的导入路径和库引用模式。如果当前工程在GOPATH环境，那么检查是否有DM工具配置文件。然后通过查找Vendor目录中的维护库信息来确定相应的库是否被删除或重新定位。
  2. 收集下游项目信息：利用Github的APIcode_search_url。简单来说，查找当前项目go.mod的需求指令和源文件中的导入指令或者DM工具的配置项来查找下游项目。HERO收集每一个下游项目的最新版本，其余信息的收集同上。
  3. 收集上游项目信息：
    1. 若当前项目为Go Modules，则通过当前项目的导入路径和版本信息来收集上游项目
    2. 若当前项目为GOPATH，则通过当前项目的导入路径和源文件来收集上游项目

检测类型

  • Type A：module-unaware项目PvP_vP​v​​在Go Modules里引用了上游项目upaup_aup​a​​的特定版本，而另一个上游项目upbup_bup​b​​则使用了最新版本，主要使用了major branch策略（主要分支），发布了一个v2+版本

    • PvP_vP​v​​引用upaup_aup​a​​旧版本：不会发生build错误
    • PvP_vP​v​​更新upaup_aup​a​​以引用最新版本：无法识别upbup_bup​b​​的虚拟导入路径

  • Type B.1：基于Go Modules的项目PvP_vP​v​​通过一个基于GOPATH的module-unaware项目upaup_aup​a​​间接引用了一个上游项目upbup_bup​b​​的v2+版本。由于两种方式对于导入路径的解释不同，upaup_aup​a​​将使用upbup_bup​b​​的最新版本，而PvP_vP​v​​仍将是用upbup_bup​b​​的旧版本

  • Type B.2：基于GOPATH的项目PvP_vP​v​​引用了仅在其Vendor目录里维护的上游项目upaup_aup​a​​（即，upaup_aup​a​​已经被删除或重定位）。

    • PvP_vP​v​​在Go Modules里没有下游项目：不会发生build错误
    • PvP_vP​v​​在Go Modules里没有下游项目：后者通过其导入路径而不是PvP_vP​v​​的Vendor目录来获取upaup_aup​a​​，则无法获取而导致build错误。

  • Type C：基于Go Modules的项目PvP_vP​v​​违反了SIV规则。一旦PvP_vP​v​​游下游项目，会直接发生build错误。
